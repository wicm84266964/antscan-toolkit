# GitHub 发布流程

本仓库采用单人维护、CI 强制验收的流程。日常开发不直接推送 `main`。

## 修改代码

1. 从最新 `main` 创建一个主题分支。
2. 在主题分支完成修改，并运行 README 中与改动相关的本地测试。
3. 检查提交清单，确认没有下载的标本、STL、渲染图、数据库、凭据或运行输出。
4. 推送主题分支并创建 Pull Request。
5. 等待必需的 GitHub CI 全部变绿，然后再合并。

本仓库只有一名主要维护者，不要求第二个人批准；CI 通过是不能跳过的批准条件。

## 发布版本

1. 确认 Pull Request 已合并，并且 `main` 最新提交的 CI 为绿色。
2. 在 GitHub 的 Actions 页面选择 `Create release`。
3. 选择 `main`，填写一个从未使用过的版本号，例如 `v1.0.0`。
4. 工作流会复核 CI、创建不可变 tag，并生成 GitHub Release。

不要手工创建、移动或覆盖公开 tag。真实网络下载和 Blender 实机测试属于发布前人工验收，不进入每次 CI。
